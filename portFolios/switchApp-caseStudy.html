SwitchApp Web Platform for SMEs and Retailers

End-to-end design for a real fintech web platform used by SMEs and retail merchants to manage money workflows. The focus was clarity, trust cues, and consistent UI patterns across payments and account actions. Mobile is not shipped yet, web only.

Type: Real product Platform: Web Users: SMEs and retail merchants Work: UX, UI, system patterns, developer handoff specs

Context

What the platform is, who it serves, and what needed to improve.

What SwitchApp is

SwitchApp is a fintech web platform that helps SMEsand retail merchants handle day-to-day money operations. The platform supports payment collection and merchant actions across multiple modules while keeping interactions predictable and trustworthy.

Why this mattered

Money workflows have low tolerance for confusion. If hierarchy is weak or modules behave differently, users hesitate, make mistakes, or abandon tasks. The goal was a web experience that feels consistent, clear, and reliable across core merchant journeys.

My role

What I owned and what I delivered.

Ownership

I owned the end-to-end product design for the web platform: UX flows, UI execution, system-level patterns, and detailed developer handoff specifications.

Handoff

I provided handoff specs to support implementation: component usage rules, spacing and layout guidance, state definitions, and interaction notes for key flows.

Key modules designed

All major merchant workflows covered on web.

Payments and payment links

Designed flows to collect payments through payment links with clear creation, sharing, and tracking. The interface makes it obvious what the merchant is generating and what the customer will experience.

Products with payment links

Designed product setup so merchants can attach payment links to products, manage pricing, and use the same collection patterns without relearning the UI.

Top up and transfers

Designed money movement flows with strong hierarchy and confirmation: who, amount, fees if applicable, and what happens next. Error states were built to be recoverable.

Payouts and bill payments

Designed payout and bill payment flows that follow the same interaction patterns as transfers, so the platform feels consistent across different transaction types.

Approach

How the web platform was made consistent and trustworthy.

Flow first

Mapped the core journeys across modules so UI patterns reflect real actions and rules, not just visuals. This reduced inconsistencies between payment, transfer, and bill flows.

Trust cues and hierarchy

Strengthened hierarchy, confirmation, and state messaging so merchants can confidently complete high-stakes actions. The platform communicates clearly at every step.

System patterns

Established reusable patterns for forms, tables, buttons, and status messaging to support scalability. These patterns are ready to extend into additional modules and future mobile work.

System states

Designed intentionally so the UI never goes silent.

Transactional states

  • Success states that confirm completion and show next options
  • Pending states that explain what is happening and what to expect
  • Failed states with clear reasons where possible and a recovery path

Empty and loading states

  • Empty states that guide merchants toward the next meaningful action
  • Loading states that keep users informed and reduce uncertainty
  • Consistent messaging tone across modules

Outcome

Clear outcomes without fake metrics.

Result

A cohesive web platform designed for SMEs and retailers, with consistent UI patterns across payments, payment links, top ups, payouts, transfers, bill payments, and product-linked collections.

Next

Extend the system into more modules and translate validated web patterns into a mobile experience when the roadmap supports it.
